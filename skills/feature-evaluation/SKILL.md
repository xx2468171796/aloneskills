# 功能评估技能 (Feature Evaluation Skill)

在开始编写任何代码之前，必须执行此评估流程。

## 评估流程

### Step 1: 意图分析

回答以下问题：

1. **任务类型**：UI 还是逻辑？
2. **涉及本地操作**：是否需要文件系统/Shell？
   - 是 → 检查是否有桌面端支持
3. **是否需要设计文档**：
   - 新功能/能力 → 建议创建设计文档
   - Bug 修复/Typo → 直接修复

### Step 2: 技术约束检查

根据项目技术栈检查：

| 检查项 | 常见规则 |
|--------|------|
| 后端逻辑 | Server Actions vs API Routes |
| UI 组件 | 组件库优先 vs 原生 HTML |
| 样式 | Tailwind CSS vs CSS 文件 |
| 状态管理 | 客户端状态 vs 服务端状态 |
| 数据库 | ORM 优先 vs 直接 SQL |
| 类型安全 | TypeScript Strict |

### Step 3: 架构原则检查

- [ ] **原子化设计**：组件是否拆分合理？
- [ ] **单一职责**：UI 组件是否只负责展示？
- [ ] **DRY**：是否有重复逻辑需要抽离？
- [ ] **YAGNI**：是否只实现当前需求？
- [ ] **KISS**：实现是否足够简单？

### Step 4: 能力匹配

根据任务类型，调用对应的技能规范：

| 任务类型 | 调用技能 |
|----------|----------|
| 登录注册相关 | `auth` |
| UI 组件开发 | `ui-components` |
| 权限控制 | `rbac` |
| 数据库操作 | `db-operations` |
| 构建部署 | `build-standards` |
| 测试相关 | `testing` |
| UI/UX 设计 | `ui-ux-pro-max` |

## 设计文档决策树

```
收到开发请求
├─ Bug 修复（恢复预期行为）？ → 直接修复
├─ Typo/格式/注释？ → 直接修复
├─ 依赖更新（非破坏性）？ → 直接修复
├─ 新功能/能力？ → 考虑创建设计文档
├─ 破坏性变更？ → 需要设计文档 + 迁移方案
├─ 架构变更？ → 需要设计文档
└─ 不确定？ → 先与团队讨论
```

## 预开发检查清单

```markdown
## 开发前检查

- [ ] 已阅读项目约定文档
- [ ] 已检查相关规范
- [ ] 已确认不违反技术约束
- [ ] 已选择正确的技能规范
- [ ] 已确认是否需要设计文档

## 技术约束确认

- [ ] 遵循项目的后端模式
- [ ] 使用项目的组件库
- [ ] 使用项目的样式方案
- [ ] 使用项目的验证库
- [ ] 使用项目的数据库访问模式
- [ ] 使用环境变量存储敏感信息
```

## 变更影响评估

对于重大变更，必须评估：

1. **影响的模块**：哪些文件/模块受影响？
2. **破坏性变更**：是否影响现有 API/Schema？
3. **迁移方案**：如何处理历史数据/兼容性？
4. **回滚方案**：如何回滚变更？
